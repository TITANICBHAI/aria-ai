Technical Feasibility and Architectural Framework for On-Device Autonomous Agents on Mid-Range Android Hardware
The pursuit of autonomous digital agents capable of operating entirely on-device, without reliance on cloud infrastructure, represents one of the most ambitious engineering frontiers in contemporary mobile computing. For a device characterized by the mid-range Samsung Galaxy M31—equipped with the Exynos 9611 chipset, an ARM Mali-G72 MP3 GPU, and 6GB of LPDDR4X RAM—the challenges of implementing a "learning-from-zero" AI system are multifaceted. This report provides an exhaustive technical analysis of the feasibility and implementation strategies for such a system, utilizing a polyglot monorepo structure consisting of a JavaScript-based user interface and a Kotlin-driven logical core. Central to this architecture is the deployment of Llama 3.2 1B as a reasoning engine, integrated with Android’s Accessibility APIs, MediaProjection for vision, and Low-Rank Adaptation (LoRA) for continuous on-device learning and fine-tuning.
Monorepo Architecture for Polyglot Android Development
The complexity of a system requiring high-level UI flexibility and low-level logical performance necessitates a robust, unified development environment. A monorepo is defined as a single repository containing multiple projects, assets, and shared packages, allowing for predictable dependency management and reduced integration overhead. In a project where the frontend is developed in JavaScript (via React Native) and the "brain" is developed in Kotlin, a monorepo facilitates the synchronization of the two ecosystems.
Structural Foundations and Tooling Selection
The selection of monorepo orchestration tools is critical for managing the divergent build requirements of JavaScript and Kotlin. While Yarn Workspaces provide a basic mechanism for sharing node_modules, advanced platforms like Nx and Turborepo offer enhanced capabilities such as dependency graph analysis and distributed caching. Nx, often described as a "full-blown development platform," is particularly suited for large-scale enterprise applications and complex ecosystems that require rigorous standards.
A recommended project structure for this polyglot agent separates the consumer-facing applications from the internal logic modules. This separation maintains clear ownership boundaries, where applications depend on packages, but packages never depend on applications, ensuring a predictable dependency graph.
| Directory | Content Type | Technical Purpose |
|---|---|---|
| apps/mobile-agent/ | React Native (TS/Kotlin) | The primary Android application containing the bridge and manifest. |
| apps/web-dashboard/ | React / Next.js | A local web-based interface for monitoring agent logs and RL metrics. |
| packages/brain/ | Kotlin / C++ | Core AI logic, LLM inference wrappers, and JNI implementations. |
| packages/learning/ | Kotlin / DL4J | Reinforcement learning modules and LoRA fine-tuning logic. |
| packages/ui-core/ | TypeScript / React | Shared UI components used across the mobile app and web dashboard. |
| packages/shared-utils/ | TypeScript / JS | Common utility functions for data formatting and configuration. |
Dependency Resolution and Hoisting Mechanics
A significant technical hurdle in mobile monorepos involves the conflict between JavaScript package managers and the Android Gradle build system. Traditionally, React Native and its Metro bundler assume a flat project structure where all dependencies are located in a local node_modules directory. In a monorepo, package managers like Yarn or PNPM "hoist" shared dependencies to the repository root to prevent duplication.
For the Samsung M31's build process to function, the Android Gradle configuration must be manually adjusted to resolve dependencies from the hoisted root. This involves updating settings.gradle and build.gradle files to point reactNativeDir and codegenDir toward the workspace root rather than a relative local path. Furthermore, when using PNPM, the node-linker=hoisted setting in .npmrc is recommended to avoid mysterious compilation errors in the Kotlin native modules, ensuring the autolinking mechanism correctly identifies the required libraries.
The JSI Bridge and High-Performance Logic
The interaction between the JavaScript UI and the Kotlin logic layer must be optimized for a mid-range processor like the Exynos 9611. While the legacy React Native bridge relied on asynchronous JSON serialization, which introduced latency, the New Architecture utilizes the JavaScript Interface (JSI). JSI allows the JavaScript engine (Hermes) to hold direct references to Kotlin/C++ host objects.
For an AI agent, the Kotlin logic layer acts as the orchestrator for the LLM. Implementing this as a TurboModule ensures that when the UI sends a task to the agent, the call is executed with near-native performance. The Kotlin side manages high-performance threading, utilizing Coroutines to prevent blocking the main UI thread during intensive inference tasks. This architecture allows the agent's "brain" to process multimodal input and dispatch accessibility actions with minimal overhead.
On-Device Reasoning with Llama 3.2 1B
The reasoning capabilities of the agent are anchored by the Large Language Model. For the Samsung M31, which features 6GB of total system RAM, the selection of the model is constrained by the Resident Set Size (RSS) memory limits. Llama 3.2 1B emerges as the optimal choice due to its specific design for mobile edge devices and its superior "Vision-to-Action" potential compared to logic-heavy but vision-limited models like Phi-3.
Architecture and Efficiency of the 1B Model
Llama 3.2 1B is a foundational, decoder-only transformer model developed by Meta, optimized for execution on resource-constrained environments. It features an optimized architecture that employs Grouped-Query Attention (GQA) to enhance inference scalability by reducing memory bandwidth usage for key and value tensors. The model was distilled from larger Llama 3.1 models (8B and 70B), allowing it to retain a significant degree of the reasoning performance of its larger siblings despite its 1.23 billion parameter count.
The model supports a context length of up to 128,000 tokens, which is unprecedented for a model of this scale. This expanded context is vital for an agent that must remember its recent interactions with the UI and process complex multi-page document information. However, the Exynos 9611 hardware dictates a practical limit on active context; memory usage grows quadratically with sequence length, making the full 128K window unfeasible for simultaneous UI and AI operations on a 6GB RAM device.
Memory Constraints and Quantization Strategy
Total system RAM on the M31 is a shared resource. After the Android OS and system services consume their baseline, approximately 2.5GB to 3.5GB may be available for application use. A Llama 3.2 1B model in 16-bit precision (BF16) requires roughly 2.3GB of VRAM. This leaves virtually no headroom for screen capture, OCR, or the application UI.
| Quantization Level | Model Size (MB) | Resident Memory (RSS) | Inference Speed (M31 est.) |
|---|---|---|---|
| BF16 (16-bit) | ~2358 | ~3185 MB | < 2 tokens/sec |
| Q4_K_M (4-bit) | ~870 | ~1,500 - 1,900 MB | ~10-15 tokens/sec |
| IQ2_S (2-bit) | ~581 | ~900 - 1,100 MB | ~20+ tokens/sec |
For the M31, 4-bit quantization (Q4_K_M or Q4_0) is the recommended path. This strikes a balance between reasoning quality and performance, allowing the model to stay within the RAM limits while leaving space for active screen buffers. More aggressive quantization, such as 2-bit (IQ2_S), significantly reduces the model's accuracy and IFEval (Instruction Following Evaluation) scores, making it unsuitable for complex navigation tasks.
Inference Frameworks: llama.cpp vs. MediaPipe
There are two primary paradigms for running Llama 3.2 on Android: the community-driven llama.cpp and the Google-native MediaPipe LLM Inference API.
llama.cpp is a C++ implementation that leverages ARM NEON extensions for hardware-accelerated matrix multiplication on mobile CPUs. It is highly customizable and can be integrated into the monorepo via the Android NDK and JNI. llama.cpp is advantageous for developers who require fine-grained control over memory mapping (use_mmap = true) and GPU offloading via Vulkan.
The MediaPipe LLM Inference API, built on LiteRT, provides a more streamlined, higher-level interface. It is specifically designed to leverage the GPU and the Android Neural Network API (NNAPI) for acceleration. For an agent that requires multimodal vision, MediaPipe's ability to handle vision and text inputs through its LlmInferenceSession makes it a strong candidate for "Vision-to-Action" workflows.
Multimodal Perception: Screen Reading and OCR
To learn and execute tasks autonomously, the agent must perceive the device's state. This is achieved through "semantic screen reading," which fuses raw visual data with the structural metadata provided by the Android OS.
Screen Capture via MediaProjection
The agent's visual perception begins with the MediaProjection API, which allows it to capture the contents of the display. Implementing this requires a foreground service and explicit user consent, which provides a security token to create a VirtualDisplay.
The Exynos 9611’s memory bandwidth is a limiting factor for high-resolution screen capture. To maintain performance, screenshots must be projected onto a Surface and then downsampled before being processed by the AI modules. In practice, a resolution of 512x512 or 224x224 pixels is often sufficient for character recognition and UI element detection, significantly reducing the computational load on the Mali-G72 GPU.
Optical Character Recognition and White-Space Alignment
Once a screenshot is obtained, Google’s ML Kit OCR engine is used to extract textual information. Advanced agent implementations do not simply extract a bag-of-words; they convert the screenshot into "white-space structured text". This technique maintains the relative position of text nodes, allowing the LLM to understand visual groupings, such as a label appearing above an input field or a price listed next to a product image.
Semantic Interpretation of the Accessibility Tree
While OCR provides a visual map, the Android Accessibility Service provides a structural map. This service allows the app to traverse the UI hierarchy (the node tree), identifying every interactable element, its class (e.g., android.widget.Button), and its resource ID.
The agent transforms this complex XML tree into a semantic text summary that is "LLM-friendly." By assigning unique IDs (e.g., [#4] or ``) to interactable elements and pairing them with their textual labels, the agent creates a compressed representation of the screen that the Llama 3.2 model can evaluate to decide the next action.
The Action Layer: Accessibility APIs for Autonomous Interaction
Execution of the agent's decisions is handled by the AccessibilityService. This system-level capability allows the agent to simulate human interactions—such as taps, swipes, and text entry—across any installed application.
Gesture Dispatch and Real-Time Execution
The agent communicates its intent via tool calls. If the reasoning engine decides to click a button, the system resolves the semantic ID of that button to physical X/Y screen coordinates using the data from the accessibility node tree. The AccessibilityService.dispatchGesture method is then used to execute a GestureDescription, which the Android OS treats as a physical touch event.
This allows the agent to perform actions such as:
 * Sequential Clicks: Navigating through settings menus.
 * Long Presses: Triggering context menus in third-party apps.
 * Swipes: Scrolling through long documents or playing gesture-based games.
 * Text Injection: Entering code into an IDE or drafting messages in a communication app.
Task-Oriented Feedback Loops
The autonomous loop is not a one-way command sequence but a feedback-driven cycle. Every action's result—whether it was a success, a failure, or a transition to a new screen state—is fed back into the LLM's context for the next turn. This "multi-turn memory" ensures the agent knows which moves worked and can recover from errors, such as a button that didn't respond or a pop-up that blocked its path.
Learning from Zero: On-Device Fine-Tuning and RL
The most ambitious component of the agent is its ability to learn from zero, improving its performance without external server intervention. This is achieved through Parameter-Efficient Fine-Tuning (PEFT) and reinforcement learning algorithms adapted for the edge.
Parameter-Efficient Fine-Tuning with LoRA
Low-Rank Adaptation (LoRA) is the standard for fine-tuning large models on hardware with limited VRAM. Traditional fine-tuning requires updating all parameters of a model, which for a 1B model would involve massive computational and memory costs. LoRA, instead, freezes the base model weights and inserts small, trainable rank-decomposition matrices into each transformer layer.
During training, only these lightweight matrices (often representing less than 1% of the total model size) are updated. This allows the agent to learn new domains—such as a specific programming language or a specialized UI pattern—using a dataset of only a few hundred megabytes.
In this mathematical representation, W_0 is the frozen pre-trained weight matrix, and A and B are the low-rank matrices being trained. For the M31, this technique is the only viable path to on-device training, as it lowers VRAM requirements to approximately one-eighth of that required for full training.
Reinforcement Learning and Game Playing
To learn how to play games or navigate novel apps from zero, the agent utilizes a Reinforcement Learning (RL) module. The Android environment is exposed to the agent as an RL workspace, where it receives pixel observations (state) and rewards (feedback) based on its actions.
The REINFORCE algorithm (policy gradient) is effective for this purpose. The agent's policy—represented as a neural network—is trained to maximize cumulative rewards. For instance, in a game, the agent might receive a positive reward for an increase in score and a negative reward for a "game over" signal.
| RL Component | Android Agent Implementation |
|---|---|
| Agent | The Kotlin-based policy network (PolicyGradient) |
| Environment | The Android OS and the current foreground application |
| State | Downsampled RGB pixel values and OCR text summaries |
| Action | Localized touch, swipe, and lift events |
| Reward | Success notifications or changes in UI state (e.g., progress bar advancement) |
The agent can also engage in Inverse Reinforcement Learning (IRL), where it "watches" YouTube videos to learn optimal policies. By running OCR on video frames and observing the sequence of UI states and actions taken by a human expert, the agent extracts optimized values that serve as pre-training data for its own policy networks.
Technical Feasibility Assessment on Exynos 9611
The technical viability of this autonomous system depends heavily on the specific architecture of the Exynos 9611 SoC. Released in late 2019, this 10nm chip utilizes a big.LITTLE core configuration.
Processing and Computational Power
The Exynos 9611 integrates four Cortex-A73 performance cores (2.3 GHz) and four Cortex-A53 efficiency cores (1.7 GHz). AI inference and training tasks will be distributed across these clusters. While the A73 cores are capable, they lack the specialized NPU (Neural Processing Unit) performance of modern flagships.
Benchmarking of 1B models on similar mid-range hardware suggests an inference speed of approximately 8 to 12 tokens per second for generation tasks. While this is significantly slower than the 50-70 tokens per second seen on the Snapdragon 8 Elite, it is functionally sufficient for an autonomous agent. An agent does not need real-time conversational speeds; it can afford a few seconds of "thinking time" to analyze a screenshot and decide on an action.
Thermal Management and Sustained Performance
Intensive tasks such as LoRA fine-tuning and high-frequency screenshot processing generate significant heat. Mid-range devices like the M31 are prone to thermal throttling, where the system reduces clock speeds to prevent hardware damage.
To mitigate this, the architecture must implement "sustained performance mode" (Window.setSustainedPerformanceMode()) and automatic execution pausing. The learning modules should be designed to run as background tasks that throttle based on the device's battery temperature. Furthermore, using a "training-data -> optimized value" approach—where the agent collects data during usage and performs training during periods of external charging—is the most feasible strategy for long-term improvement without degrading user experience or battery health.
Integration Roadmap and Systematic Workflow
The construction of the agent follows a systematic workflow that integrates perception, logic, and action into a continuous autonomous loop.
1. The Observation Phase
The Kotlin logic layer triggers the perception module.
 * Visual Capture: MediaProjection takes a screenshot.
 * Structural Capture: AccessibilityService dumps the UI node tree.
 * OCR Engine: ML Kit extracts text from the visual capture.
 * Synthesizer: The structural tree and OCR text are merged into a unified semantic map.
2. The Reasoning Phase
The semantic map, along with the user's high-level goal (e.g., "Find a tutorial on how to code in Kotlin"), is sent to the Llama 3.2 1B model.
 * Prompting: The model is prompted with a system message describing its tools (Click, Swipe, Type).
 * Inference: The model evaluates the screen: "There is a search bar. Action: Type 'Kotlin tutorial' into the search bar".
 * JSON Generation: The model outputs the decision in a structured format: { "tool": "TypeTool", "text": "Kotlin tutorial", "node_id": "search_input" }.
3. The Execution Phase
The Kotlin backend parses the model's decision.
 * Coordinate Resolution: The GestureEngine identifies the pixel coordinates for search_input.
 * Dispatch: AccessibilityService.dispatchGesture simulates the touch event.
 * Verification: The system waits for the screen to update and verifies if the text was successfully entered.
4. The Learning Phase
If the task is novel, the agent initiates its learning loop.
 * Data Collection: The sequence of screen states and actions is recorded.
 * LoRA Update: The model performs a local fine-tuning step using the successful execution trace.
 * RL Refinement: If the action resulted in a higher score or goal achievement, the reward signal updates the policy weights.
Conclusion and Engineering Recommendations
The development of an on-device autonomous agent for the Samsung Galaxy M31 is technically feasible, albeit with strict hardware-imposed constraints. The architecture must prioritize memory efficiency and thermal management over real-time latency.
The monorepo structure, utilizing Nx and a hoisted Gradle configuration, is the only sustainable way to manage the polyglot codebase. The "brain" of the system must be implemented in Kotlin, utilizing the JSI for high-performance communication with the JavaScript UI.
Llama 3.2 1B is the definitive choice for the reasoning engine. Larger models like the 3B variant or Phi-3 are too memory-intensive for a 6GB RAM device when combined with the overhead of vision and accessibility services. 4-bit quantization is mandatory to ensure the model remains in RAM alongside screen buffers.
For perception, a hybrid approach using ML Kit OCR and the Android Accessibility node tree provides the most robust semantic understanding of the screen. Continuous learning must be constrained to LoRA fine-tuning and policy-gradient reinforcement learning to ensure the training process is computationally viable.
Ultimately, the feasibility of this system hinges on the "collect-then-train" strategy. By separating data collection (during active use) from model training (during idle charging), developers can overcome the performance limitations of the Exynos 9611. This creates a self-improving, entirely private agent that represents a significant leap forward in on-device AI maturity.
