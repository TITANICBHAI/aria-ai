# Architecture Overview

For a hybrid Kotlin/JS Android app with embedded AI, a monorepo can unify code and build processes. Modern tooling (e.g. Nx, Yarn workspaces) supports mixing Kotlin (Android or backend) and JavaScript (UI) projects in one repository【38†L42-L45】【21†L347-L356】. For example, an Nx “Gradle Workspace” can host a Kotlin/Jetpack Compose Android app or Spring Boot service alongside a React/TypeScript UI. Developers can run `nx serve backend` (Kotlin service) and `nx serve frontend` (JS app) in parallel【38†L204-L212】. Alternatively, React Native allows a JS-based UI with Kotlin/Java “Native Modules” for business logic – each Kotlin class extends `ReactContextBaseJavaModule` and exposes methods to JS via `getName()`【41†L153-L161】【41†L173-L180】. In practice, the monorepo is structured with separate folders (e.g. `apps/android`, `apps/web`, `packages/*`) and shared libraries; this enables shared types, consistent versioning, and unified CI/CD across the Kotlin and JS layers【21†L347-L356】【38†L204-L212】.

# On-Device LLM Selection (Llama 3.2 vs Phi-3)

**Llama 3.2 (Meta):** Meta’s Llama 3.2 series includes *1B* and *3B* parameter models specifically designed for edge/mobile use【5†L70-L79】【5†L111-L116】. These smaller LLMs have multimodal training (vision+text) and support image understanding. For example, the 1B variant is marketed as “ultra-lightweight” for on-device and edge deployment【5†L70-L79】【5†L111-L116】. In practice, a Llama 3.2 1B Instruct model quantized to 4 bits (Q4_K_M) is only ~808 MB on disk【13†L114-L122】, fitting comfortably in a 6 GB RAM device. This model can analyze screenshots and UI images (vision-to-action) and excel at accessibility tasks (describing UI elements) thanks to its multimodal vision training. Its 128K context length and multilingual support also help with summarization and retrieval use cases【5†L80-L87】.

**Phi-3 (Microsoft):** The Phi-3 mini model is about 3.8B parameters【10†L75-L83】. It is tuned for reasoning and logic (math, code) in constrained environments. Microsoft reports Phi-3 Mini “outperforms models twice its size” on reasoning benchmarks【10†L75-L83】. However, the Mini’s larger parameter count makes it heavy in RAM on mobile. Even compressed, a 3.8B model (GGUF) would occupy multiple gigabytes and compete with the OS. Phi-3 is ideal for logic/math tasks, but *not* optimal for real-time screen/OCR tasks on 6 GB hardware. In short: **use Llama 3.2 (1B/3B) for screen and accessibility** – it’s built for mobile vision, while **use Phi-3 only for local reasoning tasks** if at all.

# Android On-Device AI Frameworks

- **MediaPipe Tasks & LLM Inference API:** Google’s MediaPipe (AI Edge) provides a ready-made Android API (`com.google.mediapipe:tasks-genai`) to run LLMs on-device【29†L13-L21】. The LLM Inference Task lets an Android app load an LLM (in a `.task` file) and generate text, answer questions, or summarize, entirely offline【29†L13-L17】. It even supports multi-modal prompts (text+image or audio) by enabling vision/audio input flags in the configuration【29†L60-L64】【29†L79-L87】. MediaPipe’s AI tasks are optimized for newer devices (Pixel 8 / S23 and up) and GPU/NPU acceleration, but in principle can run on midrange hardware if properly quantized. Notably, **LiteRT-LM** is Google’s newer runtime for on-device LLMs with Kotlin support and hardware acceleration (GPU/NPU)【35†L25-L33】. LiteRT-LM supports multimodal pipelines and model tools, and is compatible with the llama.cpp GGUF format. (In practice, one would add a Maven/Gradle dependency and call `LlmInference.createFromOptions(...)` to load the model and run inference.)
  
- **Google AI Edge (Gemini Nano, ML Kit):** Google’s AI Edge portal highlights on-device models. For example, **Gemini Nano** is Google’s mini multimodal LLM (part of Gemini 1.x) made for mobile. It’s available via Android’s AICore system service (as of Oct 2024 on Pixel 9) and used in Google apps (e.g. TalkBack for image description)【24†L140-L148】. While not open-source, it exemplifies the capability: Gemini Nano handles text/image prompts and runs fully on-device. For custom solutions, Google’s ML Kit offers a suite of on-device vision and text APIs. ML Kit’s *Text Recognition v2* can perform OCR on the device, extracting screen text with high accuracy【43†L65-L73】【43†L167-L173】. ML Kit runs offline (no cloud) and is optimized for real-time camera input【43†L65-L73】. In summary, Android provides components (MediaPipe + ML Kit + AICore) to assemble an on-device “AI assistant”: MediaPipe LLM tasks to run LLM inference, ML Kit for OCR, and Accessibility or MediaProjection for screen capture.

# Monorepo Architecture (Kotlin + JS UI)

A monorepo allows code sharing and unified deployment. For example, Nx’s Gradle plugin lets you combine a Kotlin service or Android app with a React/TypeScript UI in one workspace【38†L42-L45】【38†L75-L84】. In this setup, the Kotlin code (backend or mobile logic) can expose REST APIs or shared libraries consumed by the JS UI. Commands like `nx build` or `nx test` then work across both domains【38†L204-L212】. Similarly, a Yarn or pnpm workspace can contain an `apps/` directory for multiple frontends and a `libs/` directory for shared code【21†L347-L356】. In practice, you might have: `apps/mobile` (React Native), `apps/web` (Next.js), plus shared packages (UI components, utilities). Kotlin logic (e.g. Android Services or Spring Boot) lives in its own directory (e.g. `apps/android-backend` or `apps/android-logic`), and the JS UI in `apps/ui-react`. Using React Native, you bridge to Kotlin via Native Modules【41†L153-L161】. Each Kotlin native module class overrides `getName()` and methods annotated with `@ReactMethod` become callable from JavaScript【41†L173-L180】. This lets the JS UI trigger Android functions (like invoking the AI model or interacting with Accessibility).

# On-Device Learning: Fine-Tuning and Reinforcement

True on-device training is **limited** by hardware. However, parameter-efficient fine-tuning (LoRA) is supported by some frameworks. Google’s MediaPipe LLM Inference API explicitly supports LoRA adapters for certain models (Gemma-2, Phi-2)【29†L97-L105】. In practice this means you can precompute a small set of weight matrices (e.g. for a new prompt style) and load them at runtime to adapt the base model. This *low-overhead fine-tuning* is feasible on-device with appropriate tools, though it’s still experimental. Full retraining (even LoRA) on a phone is generally impractical beyond small updates.

Reinforcement Learning (RL) on the device is even heavier. A fully LLM-based RL agent (learning from scratch) is unrealistic on 6 GB RAM. Instead, one might incorporate a *dedicated RL module* for narrow tasks. For example, a lightweight custom agent could learn simple game strategies or UI workflows offline (or in short bursts) and then run locally. The LLM would *not* itself train via RL; rather, it provides strategy hints or explanations in natural language. For instance, an LLM can parse a game screen and suggest moves (“A 'Play' button appears now, tap it”), while the actual control logic could be a separate rule-based or small RL system. In short: **LLMs power reasoning and language guidance, while any hard RL control would be a distinct module.**

# Implementation Checklist

- **Model choice & quantization:** Use **Llama 3.2 1B** (or 3B) quantized to 3–4 bits (GGUF format via llama.cpp). A 1B model at Q4_K_M is ~800 MB【13†L114-L122】, fitting in 6 GB RAM with headroom. **Never use a full (32-bit) model** – always compress (4-bit or lower). The GGUF binary format is native to llama.cpp【46†L91-L99】. For llama.cpp integration, convert Meta’s checkpoints (e.g. `meta-llama/Llama-3.2-1B-Instruct`) to GGUF with `quantize` script, or download community 4-bit dumps.

- **Model loading (llama.cpp):** Include llama.cpp in the Kotlin project via JNI (or use the llama.cpp Android build) to load and run inference on the GGUF model. Alternatively, use MediaPipe’s LiteRT-LM: add the `litert-lm` Gradle dependency and point `LlmInferenceOptions.setModelPath(...)` at the GGUF file (on internal storage).

- **Screen capture:** Implement an **Android AccessibilityService** or **MediaProjection**. AccessibilityService can capture UI `AccessibilityNodeInfo` and screenshots of the app screen. For games/secure content, MediaProjection (with user consent) can capture the display. Convert the captured image (Bitmap) to an `MPImage` or ByteBuffer. Pass this image to the LLM inference API by enabling the vision modality (e.g. `LlmInferenceSession.LlmInferenceSessionOptions.setEnableVisionModality(true)`【29†L79-L87】). If just reading text, first run ML Kit’s OCR on the bitmap (fully on-device) to get text strings.

- **LLM inference (accessibility):** Once an image and/or text snapshot is captured, invoke the LLM to *describe* the screen. For example, feed the screenshot into LLM Inference: `"Describe the UI elements visible in this image."` The model (trained on multimodal data) will output: *“There is a ‘Play’ button in center, a settings icon top-right”*. Return that text via TextToSpeech or UI.

- **Performance optimizations:** On Exynos 9611 with 6 GB RAM, keep total footprint small. Use Llama 3.2 1B with 4-bit or 3-bit quantization. Avoid loading >1.5 GB of model. If using Llama 3.2 3B, only do so if the app isn’t also running games/video. Leverage NPU/GPU if available (MediaPipe LLM or LiteRT-LM can use NNAPI or GPU). Use input token limits (~512) to reduce inference time. Perform heavy tasks (image processing) off the main UI thread. 

- **Continuous Learning (LoRA):** If adaptive learning is needed, periodically collect user corrections (e.g. user says “no, it’s not Settings, it’s Share”). Offload fine-tuning to a background process using LoRA: compute adapter weights (on-device or server) and load them into the model. MediaPipe LLM supports loading LoRA weights with `LlmInferenceOptions.setLoraConfig(...)`【29†L97-L105】. This lets the model “remember” custom vocabulary or UI changes without reloading the full model.

- **Testing & tuning:** Use Google AI Edge Studio/Portal to benchmark latency and memory. The Nx/Yarn setup allows you to write shared UI code (JS) and hook it into the Android build. In development, use `nx serve` or React Native’s bundler to live-reload the JS UI, while deploying updated Kotlin components via Gradle. Validate on-device performance (benchmark token/sec). If inference is slow, consider a smaller context window or pruning. Always test on the target (Samsung M31) to ensure stability.

- **Fallback / Offload:** For tasks beyond “screen assist” (e.g. complex math, large retrieval), consider offloading to cloud or larger devices, since on-device LLMs are limited. For gaming “live AI agents,” prefer side-by-side helpers (showing strategy tips in a panel) rather than trying to run inside the game.

Each of these steps **must** respect the device’s limitations. By using Llama 3.2’s small, vision-enabled models【5†L70-L79】, quantized GGUF files【46†L91-L99】, and Android’s ML frameworks【29†L60-L64】【43†L65-L73】, a Kotlin+JS app can feasibly provide local accessibility and screen-reading AI on a midrange phone.
